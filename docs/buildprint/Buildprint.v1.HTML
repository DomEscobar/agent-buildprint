Authority rule

This is a human overview. The authoritative instructions are the snapshot files inside the Buildprint package and the runtime handoff files under .buildprint/ after agb start.

What This Contract Solves

The baseline exists because generic agent instructions fail in repeatable ways: agents over-read, stop after a checkpoint, build local MVP shells, or upgrade claims from weak evidence.

Context control

Agents get enough context for the current phase without inhaling the whole packet.

Execution pressure

Every phase needs a real vertical path, not only a plan, review note, or static screen.

Evidence honesty

Claims upgrade only when a specific command or artifact proves the matching behavior.

Required Read Order

Initial context reads must be sequential and observable. The agent must not inventory the packet, glob all markdown, or read all phase files upfront.

1. Start

BUILDPRINT.md

2. Gate

01-questions.md then 02-project-setup.md

3. Mirror

blueprint.yaml

4. Route

03-phases/phase-index.yaml then 03-phases/phase-flow.md

5. Execute

Only the active phase file selected by phase index, assignment, or local .buildprint state.

6. Prove

04-evaluation.md, evidence seed, and evidence schema before runtime evidence.

Forbidden read behavior

Do not read every phase upfront. Do not treat generated/agent-prompt.md as authority. Do not edit snapshot files to record implementation progress.

Authority Map

Buildprints deliberately separate human-readable routing, machine-readable mirrors, runtime state, and publication metadata. Mixing these layers is how packets become brittle.

Package router

BUILDPRINT.md defines the canonical packet start and invariant read order.

Setup contract

01-questions.md and 02-project-setup.md turn unknowns into defaults, blockers, or architecture decisions.

Phase routing

03-phases/phase-index.yaml selects the active and next dependency-ready phases.

Phase execution

03-phases/phase-flow.md controls plan, team gates, reviews, proof, and evidence.

Runtime truth

.buildprint/state.json, progress.md, blockers.md, and next-agent.md override stale package defaults on resumed runs.

Generated prompt is not authority

generated/agent-prompt.md is copy convenience for websites and UIs. It must point back to the packet files, and it must not become a second contract that drifts from the Buildprint.

Execution Flow

agb start creates exact snapshots and runtime state. The implementation agent then follows the packet one phase at a time. 

agb start package.json target/
        |
        v
target/.buildprint/ exact snapshots + source/state/progress/next-agent
        |
        v
agent reads .buildprint/next-agent.md
        |
        v
BUILDPRINT -> questions -> setup -> blueprint -> phase-index -> phase-flow
        |
        v
active phase only
        |
        v
plan -> team gates -> vertical path -> reviews -> proof -> evidence
        |
        v
update .buildprint/progress.md, state.json, next-agent.md
        |
        v
continue to next dependency-ready phase or stop on a real blocker

Fresh run

Use the packet read order and active_phase from phase-index.yaml.

Resumed run

Use .buildprint/next-agent.md and runtime state before stale package defaults.

Targeted replay

Use the assigned phase after confirming it exists in phase-index.yaml.

Questions And Setup Gates

The setup gate prevents agents from inventing project structure or stopping on low-impact unknowns.

01-questions.md

Ask only for high-impact unknowns: credentials, irreversible actions, paid services, destructive operations, deployment authority, product-defining forks, or security-sensitive decisions.

02-project-setup.md

Convert setup into implementation workspace alignment before phase work. Create local AGENTS.md plus .buildprint/setup.md; AGENTS.md alone is not enough.

Default behavior

Blank ordinary answers authorize AI best-judgment defaults. Missing credentials block live proof only after adapter/config/test/runtime wiring exists.

Phase Model

A phase is a proof-gated product slice. It must define product outcome, source evidence, implementation scope, interfaces, state/runtime, UX/UI, safety, quality gates, proof gate, and repair routing.

State Meaning What it does not prove
checkpoint_recorded At least one valid runtime evidence row exists for the phase. Phase completion, UX quality, or production readiness.
phase_core_passed The local vertical path works end to end with matching tests, reviews, and evidence. Live-provider, browser, deployment, or broad production qualification.
claim_qualified A specific claim has matching proof strong enough to upgrade. Unrelated claims or full-suite completion.
UI-bearing phase rule

phase_core_passed requires at least one user action path through a UI/controller boundary into runtime behavior and back into visible state or readback. Static cards, dead buttons, generic dashboards, screenshots-only proof, or review prose do not qualify.

A
Checkpoint is not completion

A ledger row proves evidence discipline. It does not prove the product slice is done.

B
Core pass is not qualification

A local deterministic path may pass while live provider, browser, deployment, or security claims remain blocked.

UI And Product Quality Gate

Browser proof cannot be only "the button exists." UI-bearing phases must prove a real user action path and reject local-MVP visuals before upgrading UX claims.

Required for UI phases

  • User action path through UI, controller, runtime behavior, and visible readback.
  • Empty, loading, error, blocked, success, and recovery states.
  • Desktop and mobile screenshot or browser evidence when tooling is available.
  • Keyboard path, focus states, disabled states, no text overlap, and adequate contrast.
  • Screenshot critique against the product's visual quality bar.

Blocking UI anti-patterns

  • Default browser controls as the final product surface.
  • Stacked forms with no workflow composition.
  • Generic cards where the domain needs graph, timeline, editor, or analysis affordances.
  • Dead buttons, no-op controls, or state cards without interaction.
  • Screenshot-only proof without repeatable user-flow evidence.

Runtime Artifacts

Runtime artifacts are written in the implementation workspace. They are not edits to the packaged Buildprint snapshots.

Before code

  • .buildprint/phase-runs/<phase-id>/plan.md
  • .buildprint/phase-runs/<phase-id>/team-gates.md

Before evidence

  • reviews/architecture.md
  • reviews/ux.md
  • reviews/qa.md
  • proof.md

Handoff and return files are optional and should appear only when real delegation to a subagent or separate specialist session happens. Do not manufacture role paperwork when the main session performs the work.

Evidence Rules

Evidence is narrow by design. Rows must prove only what their command and artifact actually exercised.

Runtime ledger

Append rows to .buildprint/evidence/evidence-ledger.jsonl. Keep packaged seed evidence immutable.

Required row fields

artifact_id, type, phase_id, status, source, proves, proof_type, provider_mode, upgrades_claim.

  • Do not set upgrades_claim: true on blockers, partial proof, missing credentials, unavailable tooling, or synthetic checks.
  • Review prose cannot upgrade implementation proof. Use separate executable proof rows.
  • Provider adapter/config tests prove seams, not live provider behavior.
  • Browser, screenshot, UX, worker, security, and data-lifecycle claims require matching proof or honest non-upgrading blocker rows.
  • One broad smoke test cannot upgrade every claim. Split evidence rows by the exact behavior exercised.

Continuation Rules

Full-suite execution continues one dependency-ready phase at a time. Later phase files are not read upfront.

Continue when

The core local vertical path passed, owned persistence is proven, required reviews/proof/evidence exist, and remaining live/browser/deployment blockers are non-upgrading qualification blockers.

Stop when

Core implementation fails, persistence is missing, local runtime proof fails, destructive/security ambiguity remains unresolved, or evidence would be fake.

Validation And Evals

Mapper OS confidence is layered. Static checks prove shape. Real replay proves downstream agent behavior. Product confidence still requires reviewing the produced outcome.

Gate Command Purpose
Packet shape npm run check:mapper-output Validates the known-good executable packet fixture.
Negative packet fixtures npm run check:mapper-output:negative Confirms obsolete or weak packet shapes fail.
Negative replay fixtures npm run check:mapper-replay:negative Rejects static UI shells, checkpoint-only evidence, UX overclaims, and browser blockers without local interaction proof.
Published audit npm run check:mapper-published Verifies published executable packets satisfy the current strict baseline.
Fast loop npm run eval:mapper-fast Combines static checks, fixture checks, golden signals, and dry-run replay.
Real agent UI smoke npm run eval:agent-small Uses interactive UI pressure against a tiny AI UI packet.
Real Local RAG npm run eval:agent-local-rag and npm run eval:local-rag-phase2 Exercises provider/chat and RAG indexing phases before expensive Swarm/MicroFish regressions.
Do not overread green checks

A green static check is not product confidence. A checkpoint-only replay is not phase completion. Browser/live/deployment qualification needs matching runtime evidence.

Anti-Patterns This Baseline Rejects

These are common failure modes that produced weak downstream implementations.

Reading everything upfront

Burns context and lets later phases pollute the current implementation slice.

Checkpoint equals completion

Evidence discipline is useful, but the phase core still needs an end-to-end vertical path.

Static UI shell

Cards and labels without a user action path do not prove a product UI.

Provider blocker as implementation escape

Missing credentials block live proof only; adapter seams, config, tests, and runtime wiring remain required.

Review prose as proof

Reviews explain risk. Executable commands and artifacts prove behavior.

Generic MVP downgrade

Full-suite mapped packets default to production-grade architecture unless scope is explicitly reduced.
