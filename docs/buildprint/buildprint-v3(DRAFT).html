Buildprint v3 / executable contract

The buildprint is code.
Proof is test results.

v2 describes what should happen in markdown. v3 makes it executable. Mode, constraints, done-when conditions, and human gates are all typed, machine-readable declarations. State is an append-only event log. Evidence is test output, not prose.

principle 01

Done-when = failing tests

Each phase declares a test file. The phase isn't done until those tests pass. No separate evidence ledger. No prose proof gate. The CI run is the ledger.

principle 02

Constraints = automated checks

Obligations from v2 role contracts become lint rules, coverage gates, or test assertions. Violation is a hard stop, not a review note.

principle 03

State = event log

No state.json to edit, no next-agent.md to go stale. Every agent action appends a row. Any agent replays the log to resume work.

01 / the file

One buildprint.ts, one truth

The buildprint is a typed TypeScript module exported as a default. It replaces BUILDPRINT.md, blueprint.yaml, every phase file, and all role contracts. The schema is the spec; the type checker is the validator. 

// buildprint.ts
import { b } from "@buildprint/v3";

export default b.project({
  name:    "acme-auth",
  mode:    "integration",         // still mode-aware — narrows type checking
  version: "3.0.0",

  // ── Phases as a typed dependency graph ──────────────────────────────
  phases: {

    "schema": b.phase({
      lens:       "data-persistence",
      outputs:    ["users table", "oauth_tokens table", "sessions table"],
      done_when:  "./done-when/schema.test.ts",
      constraints: ["migration-idempotent", "no-raw-sql-in-handlers"],
    }),

    "adapter": b.phase({
      depends_on: ["schema"],
      lens:       "integration-boundary",
      outputs:    ["ProviderAdapter interface", "GoogleOAuthAdapter", "MockAdapter"],
      done_when:  "./done-when/adapter.test.ts",
      constraints: ["all-external-through-adapter", "adapter-mockable-without-network"],
    }),

    "human:prod-keys": b.humanGate({  // ← first-class node type
      depends_on: ["adapter"],
      prompt:    "Provide Google OAuth credentials for live testing.",
      skip_if:   { env: "GOOGLE_CLIENT_ID" },
      blocks_only: ["live-proof"],       // ← doesn't block continuation
    }),

    "auth-flow": b.phase({
      depends_on: ["adapter"],           // ← not gated by human:prod-keys
      lens:       "integration-boundary",
      outputs:    ["POST /auth/google", "GET /auth/callback", "session middleware"],
      done_when:  "./done-when/auth-flow.test.ts",
      constraints: ["session-persists-restart", "wrong-token-returns-401"],
    }),

    "live-proof": b.phase({
      depends_on: ["auth-flow", "human:prod-keys"],  // ← needs both
      lens:       "integration-boundary",
      outputs:    ["live OAuth round-trip passing"],
      done_when:  "./done-when/live-proof.test.ts",
      constraints: [],
    }),

  },

  // ── Constraint definitions ────────────────────────────────────────────
  constraints: {
    "migration-idempotent":          b.test("./constraints/migration-idempotent.test.ts"),
    "no-raw-sql-in-handlers":         b.eslint("no-restricted-imports", { patterns: ["db/*"] }),
    "all-external-through-adapter":   b.coverage({ path: "adapters/**", threshold: 90 }),
    "adapter-mockable-without-network": b.test("./constraints/adapter-mock.test.ts"),
    "session-persists-restart":        b.test("./constraints/session-persist.test.ts"),
    "wrong-token-returns-401":         b.test("./constraints/auth-denied.test.ts"),
  },

});

What the type checker enforces

  • mode must be one of the eight modes.
  • lens on each phase must match the project mode's allowed lenses.
  • depends_on references must name real phase keys.
  • constraints array entries must name declared constraint keys.
  • done_when paths must resolve at build time (type-only check; runtime runs the tests).
  • humanGate nodes may appear in depends_on of other phases.
  • blocks_only must reference phase keys in the same graph.
02 / phase anatomy

What a phase declares

A phase is a dependency-ready unit of work with machine-verifiable entry conditions, outputs, done-when tests, and constraint references. No prose headings, no role-contract sections.

"adapter"
integration
lens: integration-boundary
depends_on
"schema" — must reach done before this phase is eligible.
outputs
Human-readable list of artifacts the phase is responsible for producing. Not executable — serves as context for the agent and for the log.
done_when
Path to a test file. Phase state is done when the file passes with no skipped tests. The runner is configured in package.json or buildprint.config.ts.
constraints
Keys from the project's constraints map. Each runs before the phase's done_when test. Failure blocks the phase — not just flags it.
Phase state Meaning Proof
pending Dependencies not yet done. —
eligible All depends_on phases are done. Agent may start. Event log shows all deps passed.
running Agent has emitted phase_started. Log row.
done All constraints passed and done_when tests pass with zero skips. Test runner output + log row.
blocked A humanGate dependency is unresolved and blocks_only includes this phase. Log row with gate reference.
failed Constraint or done_when test failed. Test output. Agent must repair before retrying.
Five states, no ambiguity.

There is no checkpoint_recorded-vs-phase_core_passed-vs-claim_qualified distinction. A phase is either done (tests pass, constraints pass, no skips) or it isn't. Partial progress lives in the event log; it never becomes a named state.

03 / done_when

Tests are the proof gate

The done_when file is the only thing that upgrades a phase to done. The file must pass completely — no skips — on a real execution path. This is where v2's evidence ceiling, proof tracks, and upgrades_claim logic collapse into one rule.

done-when/adapter.test.ts
// done-when/adapter.test.ts
// This file IS the proof gate for the "adapter" phase.
// It must pass with no skips — no describe.skip, no it.todo.

import { GoogleOAuthAdapter } from "../adapters/google";
import { MockAdapter }        from "../adapters/mock";
import { ProviderAdapter }    from "../adapters/types";

describe("adapter seam", () => {

  it("MockAdapter satisfies ProviderAdapter", () => {
    const adapter: ProviderAdapter = new MockAdapter();
    // type check is the assertion
  });

  it("MockAdapter resolves token without network", async () => {
    const adapter = new MockAdapter({ token: "test-tok" });
    const result  = await adapter.exchange("code-123");
    expect(result.token).toBe("test-tok");
  });

  it("GoogleOAuthAdapter is injectable as ProviderAdapter", () => {
    const adapter: ProviderAdapter = new GoogleOAuthAdapter({
      clientId: "x", clientSecret: "y", redirectUri: "http://localhost",
    });
    expect(adapter.exchange).toBeInstanceOf(Function);
  });

  // No .skip allowed — this would block the phase from reaching `done`.

});

Hard rules for done_when files

  • Zero describe.skip, it.skip, test.todo. Any skip = phase stays running.
  • No mocked network calls for things that must be tested via real paths. Live-provider tests go in a separate phase with a humanGate dependency.
  • The file exercises the exact behavior named in the phase's outputs. If an output isn't tested, the phase isn't done.
  • Runtime imports must work in CI without manual setup beyond what 02-project-setup describes.

What replaces v2 proof tracks

v2 had named tracks: live_provider_proof_blocker_only, repeatable_browser_e2e, etc. In v3:

  • Adapter/mock tests → done_when of the adapter phase.
  • Live-provider tests → done_when of a phase that depends_on a humanGate.
  • Browser e2e → done_when of a dedicated e2e phase, configured via Playwright.
  • Security denied-path → a named constraint test.
  • Persistence roundtrip → done_when of the schema/persistence phase.
Never write a done_when test that can't fail.

A test file that only checks types, or only runs mocks for what should be real behavior, allows the phase to reach done without the work being complete. Write tests that would catch a broken implementation.

04 / constraints

Obligations become automated checks

v2's role contracts contained obligations written in prose. In v3, each obligation is a named constraint with a machine-executable runner. Constraints run before every done_when test. Failure is a hard block.

Constraint typeRunnerWhat it enforcesv2 equivalent
b.eslint() ESLint rule + config Architectural invariants: no direct DB imports in handlers, no inline HTTP calls, restricted patterns. product-architect / integration-runtime prose obligations
b.test() Test runner (Vitest / Jest) Behavioral invariants: migration idempotency, session persistence, denied-path 401, adapter mockability. security-boundary / data-persistence / phase proof reviews
b.coverage() Coverage reporter Surface coverage gates: all external calls go through adapters/, threshold enforced. integration-runtime seam requirements
b.exec() Shell command Any automated check that doesn't fit the above: Lighthouse scores, bundle size, schema diff. quality gates (various) 
// constraints/auth-denied.test.ts
// Enforces: wrong-token-returns-401
// Replaces: security-boundary role contract prose

import { createTestApp } from "../test/helpers";

describe("security boundary — auth denied paths", () => {

  it("expired token → 401, not 500", async () => {
    const app = await createTestApp();
    const res = await app.inject({
      method: "GET", url: "/api/me",
      headers: { authorization: "Bearer expired-token" },
    });
    expect(res.statusCode).toBe(401);
    expect(res.statusCode).not.toBe(500);
  });

  it("missing token → 401", async () => {
    const app = await createTestApp();
    const res = await app.inject({ method: "GET", url: "/api/me" });
    expect(res.statusCode).toBe(401);
  });

  it("token of another tenant → 403", async () => {
    const app = await createTestApp();
    const res = await app.inject({
      method: "GET", url: "/api/me",
      headers: { authorization: "Bearer tenant-b-token" },
      // app configured for tenant-a
    });
    expect(res.statusCode).toBe(403);
  });

});
// buildprint.config.ts — runner config
export default {
  runner: "vitest",
  coverage: {
    provider: "v8",
    reporter: ["text", "json"],
  },
  eslint: {
    config: "./eslint.config.js",
  },
  env: {
    test: ".env.test",  // injected for all runs
  },
};
Constraints run in dependency order.

A constraint that references the database only runs after the schema phase is done. The runner resolves this from the phase graph automatically — no manual ordering.

Constraints are not advisory.

In v2, role-contract obligations could be acknowledged with a prose return note. In v3, a failing constraint stops the phase. There is no "noted, continuing anyway."

05 / human gates

Human input is a typed graph node

v2 used 01-questions.md and repair routing for human decisions. In v3, human gates are declared in the phase graph. They have dependencies, skip conditions, and explicit scope of what they block. 

"human:prod-keys": b.humanGate({
  depends_on: ["adapter"],         // waits until adapter phase is done
  prompt:    "Provide Google OAuth credentials for live testing.",
  skip_if:   { env: "GOOGLE_CLIENT_ID" },   // auto-resolves in CI
  blocks_only: ["live-proof"],              // only this phase waits
                                              // "auth-flow" can proceed
}),

"human:schema-drop": b.humanGate({
  depends_on: ["auth-flow"],
  prompt:    "You are about to DROP the legacy sessions table. Confirm? [yes/no]",
  skip_if:   { env: "BUILDPRINT_AUTO_CONFIRM_SCHEMA_DROP" },
  blocks_only: ["migration-cleanup"],
}),
🔑

human:prod-keys

External credentials gate. skip_if resolves automatically from env. Only blocks phases that need live provider behavior.

⚠️

human:schema-drop

Destructive action gate. Requires explicit confirmation. Can be pre-authorized via env in staging pipelines.

💳

human:billing-model

Product-defining fork. Agent cannot choose between per-seat vs usage pricing. Blocks the billing phase until answered.

Gate fieldPurpose
depends_onThe gate won't surface to the human until these phases are done — no point asking for credentials before the adapter is built.
promptShown to the human verbatim. Should name exactly what's needed and why. Not a place for AI best-judgment.
skip_if.envIf this env var exists and is non-empty, the gate auto-resolves. Enables CI pipelines to inject known values.
blocks_onlyOnly these phase keys wait on this gate. Other phases that don't need the gated information continue. The gate does not stall the whole build.
When to use a human gate (same rule as v2 01-questions).

Use only for decisions that are: irreversible, expensive, credentialed, destructive, or product-defining in a way the agent cannot default on. Everything else the agent decides using the mode and lens context.

06 / event log

One append-only log. Derived state only.

All runtime state is derived from .buildprint/run.jsonl. No state.json, no progress.md, no next-agent.md that can go stale. Any agent, in any session, replays the log to know where to continue.

Resuming from a new session

Agent runs agb status. The CLI replays the log, derives current state per phase, and prints which phases are eligible. Agent picks one and continues. No stale file to read.

Repair routing via the log

When a phase enters failed, the log shows the exact constraint or test that failed. Agent reads the constraint definition, fixes the code, emits phase_started attempt=N, reruns. No routing table to consult.

The log is append-only. Never edit it.

A phase_failed row is not removed when the phase later succeeds — it stays in the log as audit history. The derived status is computed from the latest event for each phase.

07 / mode & lens

Mode narrows the type system. Lens focuses each phase.

Mode-awareness from v2 survives — it still decides what kind of project this is and what obligations apply. But mode is now a TypeScript union type, and the lens is a per-phase string that the type checker validates against the project mode's allowed values.

Mode Allowed lenses done_when emphasis Dominant constraint types
product user-flow · data-persistence · ui-craft User action → controller → state → readback path passes end-to-end. No dead UI, no state-only success, real action path required.
framework primitive · composition · extension-point Primitive composes correctly; misuse states are tested. Invariant stability, no coupling to one app story.
library callable-surface · error-shape · primitive Public API surface stable; error types tested; tree-shaking safe. No breaking changes to exported signatures.
integration integration-boundary · webhook · idempotency Adapter + mock tests pass; live-provider test in separate gated phase. All external calls through adapter; sandbox/live split.
automation task-loop · approval-point · trace Plan/execute/observe loop terminates; stop conditions tested; trace emitted. Exit conditions required; no infinite loops; approval gates declared.
data-pipeline dataflow · lineage · backfill Input→transform→output chain tested on sample data; lineage provenance verifiable. Idempotent backfills; schema validation; quality thresholds.
infrastructure operations · rollback · drift Deploy applies cleanly; health check passes; rollback path tested in dry-run. No undeclared resources; drift detection enabled; permissions minimal.
mixed Any combination of above Each phase declares its own lens; done_when must match that lens's emphasis. Per-phase constraint sets; not a blanket permission to be vague.
If the type checker rejects your lens, your mode is wrong.

A phase in an integration project cannot use lens user-flow. If you need that, the project is product or mixed. The type error is the mode-classification invariant from v2, now enforced by the compiler rather than a validator script.

08 / cli

agb — the buildprint runner

The CLI is the only interface between the agent and the buildprint. It reads buildprint.ts, maintains .buildprint/run.jsonl, and executes tests and constraints.

$
agb status

Replay the event log. Print phase state table: pending / eligible / running / done / failed / blocked. Print next eligible phases. Use this at the start of every resumed session.

$
agb run [phase-id]

Start or resume a phase. Runs all constraints for the phase in dependency order, then executes the done_when test file. Appends events to the log. Errors are surfaced with constraint name and test output — not routed to a separate file.

$
agb check [constraint-id]

Run a single constraint in isolation. Useful during development to verify a specific invariant without running the full phase.

$
agb gate resolve [gate-id] [value]

Provide human input for a waiting gate. Appends gate_resolved to the log. Immediately unblocks any phase whose only pending dependency was this gate.

$
agb graph

Print the phase dependency graph as ASCII or export as graph.dot. Shows current state per node. Useful for verifying that depends_on edges match intent.

$
agb replay [--from phase-id] [--to phase-id]

Re-derive state from the log for a range of phases. Does not re-execute tests — reads the log only. Use for audit, debugging, or hand-off documentation.

$
agb typecheck

Run the TypeScript compiler over buildprint.ts. Validates mode, lenses, depends_on references, constraint keys, and done_when paths without executing anything. Run this in CI before any phase.

09 / v2 → v3

What changed, what stayed, what dropped

v3 keeps the properties that matter — mode awareness, evidence discipline, human gates, repair — and replaces the mechanisms with executable equivalents.

v2 dropped
Sequential read-order gate over 7 markdown files.
v3 insteadagb typecheck validates the whole graph in one pass. Agent reads buildprint.ts once.
v2 dropped
Six role contracts (product-architect, ux-ui-craft, …) as prose markdown under 06-contracts/.
v3 instead
Each obligation is a named constraint with a typed runner. Violation is a hard block, not a review note.
v2 dropped
Handoff / return / review artifact files per phase (plan.md, handoffs/, returns/, reviews/).
v3 instead
Constraint test files and done_when test files. If subagents are used, they run tests — not write prose returns.
v2 dropped
evidence-ledger.jsonl with upgrades_claim: true/false, proof_type, provider_mode.
v3 instead
Test results are evidence. agb run writes phase_done tests=N/N to the event log. No manual row authoring.
v2 dropped
Three-tier phase state: checkpoint_recorded / phase_core_passed / claim_qualified.
v3 instead
Five unambiguous states: pending / eligible / running / done / failed. Derived from the log.
v2 dropped
state.json, progress.md, next-agent.md, blockers.md.
v3 instead
One event log. agb status derives everything. No stale files to mislead a resumed agent.
v2 dropped
01-questions.md as a flat pre-implementation Q&A file.
v3 instead
humanGate nodes in the phase graph: depend on prior phases, auto-resolve via env, block only what they need to block.
v2 dropped
Validator script check-mapper-selected-output.mjs checking file shapes and heading tokens.
v3 instead
TypeScript compiler + agb typecheck. Mode/lens pairing, dep refs, constraint keys — all type-checked.
What stays from v2

Mode awareness

Eight modes still exist. Mode decides lens options, done_when emphasis, and dominant constraint types. Collapsing everything into product-app language is still wrong.

Evidence cannot be faked

In v2 the rule was "don't mark upgrades_claim: true on a mock". In v3 the test file either passes or it doesn't. The invariant is the same — enforcement is automatic.

Continuation ≠ qualification

A humanGate with blocks_only: ["live-proof"] still allows other phases to proceed. The live-proof phase is blocked, not failed. Other work continues.

Repair before retry

A phase_failed event means the agent fixes the code, not retries with looser tests. The constraint definition tells the agent exactly what's wrong.

One phase at a time per agent

An agent picks one eligible phase and works it to done. The graph may show multiple eligible phases — that's an opportunity for parallel agents, not a signal to batch.

Human gates for real forks only

Credentials, destructive actions, irreversible product decisions. Everything the agent can default on using mode + lens context, it defaults on. No gate for cosmetic choices.

Buildprint v3 · executable agent contract · the log at .buildprint/run.jsonl is ground truth at runtime.
