Buildprint v2 / executable contract

A Buildprint is a mode-aware execution contract for one coding agent in one phase at a time.

Each packet must classify its blueprint mode before anything else, because the mode decides phase style, outcome heading, mapped obligations, proof shape, and what counts as honest evidence. Do not force every packet into product/UI language.

  • Eight modes: product, framework, library, integration, automation, data‑pipeline, infrastructure, mixed.
  • Eight phase styles: outcome_flow, primitive_composition_map, boundary_transaction_contract, task_loop_contract, callable_contract, dataflow_contract, operations_contract, mixed_contract.
  • Six role contracts: product-architect, ux-ui-craft, integration-runtime, data-persistence, security-boundary, test-and-verification.
  • Three phase states: checkpoint_recorded < phase_core_passed < claim_qualified.
  • One ledger at runtime: .buildprint/evidence/evidence-ledger.jsonl. The packet seed ledger is immutable.

1. Read-order gate

Sequential, observable, non-batched

Initial context reads must be sequential. Do not glob the packet, batch-read all phase files, or treat generated/agent-prompt.md as authority. The agent reads exactly one phase file per run.

  1. 1

    BUILDPRINT.md — the only start point

    Reads the product brief, the canonical read order, and the implementation loop. Owns no execution state, just routing.

    Authority: routingNot a phase file
  2. 2

    01-questions.md → 02-project-setup.md

    Ask only for irreversible / expensive / credentialed / destructive / product-defining forks. Blank answers authorize AI best-judgment defaults.

    GateSetup must be explicit before any phase
  3. 3

    blueprint.yaml — machine mirror

    Holds blueprint_mode.primary, secondary, phase_style, repair_loop, and the phase list with statuses.

    Authority: validator / tooling
  4. 4

    03-phases/phase-index.yaml → phase-flow.md

    The index picks the active_phase; the flow describes phase entry, handoffs, returns, reviews, proof, and evidence rules.

    Dependency-orderedOne active phase
  5. 5

    Only the active phase’s required 06-contracts/<role>.md

    Role contracts the phase’s requires_roles: names — nothing more. Other contracts stay unread until a phase needs them.

    Scoped readAvoid context bloat
  6. 6

    The single active phase file

    Fresh run uses active_phase from the index. Resumed run uses .buildprint/next-agent.md. Targeted run uses the assignment, confirmed in the index.

    One phase at a time
  7. 7

    04-evaluation.md → 05-evidence/*

    Evaluation defines claim upgrade rules and the phase state model. Evidence schema and seed ledger are read-only context; runtime rows go to .buildprint/evidence/evidence-ledger.jsonl.

Forbidden read behavior.

Do not parallelize or batch the read order even when multi-command tooling is available. Do not read every phase upfront. Do not treat generated/agent-prompt.md as authority. Do not edit packet snapshot files to record runtime progress.

2. Mode classification — the strongest invariant

Before 02-project-setup.md / any phase

Before writing 02-project-setup.md, blueprint.yaml, or any phase, the packet must classify the selected output by blueprint mode. This is a generation invariant, not branding. Mode decides language, lens, obligations, and proof shape.

Product

Outcome flows. A user/operator can do a meaningful product operation end to end. UI usually exists.

Framework

Primitive / composition map. Reference patterns and extension points, not one app story.

Library

Callable surface and primitives. Import / API behavior, not screens.

Integration

Boundary transaction contract. Config, request/response, webhook, idempotency, sandbox vs live.

Automation

Task-loop contract. Plan/execute/observe loop, stop conditions, approval points, trace proof.

Data-pipeline

Dataflow contract. Inputs, transforms, outputs, lineage, validation, backfills, quality proof.

Infrastructure

Operations contract. Deploy/apply, health, rollback, drift, observability, permissions.

Mixed

Multi-lens packet. Each phase still names which lens it uses; not a permission to be vague.

Do not collapse modes into product-app language.

A framework is not "an app". An integration is not "a feature". An automation agent is not "a checklist". Infrastructure is not "an admin screen". If a packet looks like a product and the source is a framework, integration, or pipeline, the mapper has flattened the real shape.

3. Mode × phase style × outcome heading

Validator-enforced mapping

The mapper validator (check-mapper-selected-output.mjs) refuses packets whose phase style or outcome heading does not match the declared mode. Use this table as ground truth.

Mode Phase style Outcome heading Obligations heading Phase lens (one sentence)
product outcome_flow ## Product outcome ## Mapped product obligations User/operator does a meaningful product action end to end.
framework primitive_composition_map
callable_contract		 ## Capability outcome ## Mapped capability obligations Primitive, invariants, composition rules, extension points, misuse states.
library primitive_composition_map
callable_contract		 ## Capability outcome ## Mapped capability obligations Callable surface with stable contract, error shape, and reference patterns.
integration boundary_transaction_contract ## Capability outcome ## Mapped capability obligations External boundary: config, request/response, webhook, idempotency, sandbox/live split.
automation task_loop_contract ## Capability outcome ## Mapped capability obligations Task objective, tool/action boundary, plan/execute/observe loop, stop & approval rules.

Shared proof spine, mode-shaped.

Every phase — regardless of mode — must define: preconditions / inputs → entrypoint or use site → execution behavior → state / artifact effects → observable proof → failure / recovery → non-goals. The mode decides the language and the emphasis; the spine is invariant.

4. Packet shape

Exact files required

The executable packet has a fixed spine. Every file is required by the validator; obsolete folders (03-capabilities/, 08-evaluation/, etc.) are rejected. AGENTS.md belongs to the downstream implementation project, not the packet.

Snapshot vs runtime authority

Packet files are immutable snapshots after agb start. Runtime truth lives under .buildprint/: state.json, progress.md, blockers.md, next-agent.md, evidence/evidence-ledger.jsonl, and phase-runs/<phase-id>/….

Forbidden in the packet

START_HERE.md, PRE_IMPLEMENTATION_QUESTIONS.md, packet AGENTS.md, capability folders 03-capabilities/ through 09-evidence/, fragmented per-capability mini-files, and root manifests like CAPABILITY_INDEX.md or manifest.json.

Generated prompt is convenience copy

generated/agent-prompt.md must declare Generated from: blueprint.yaml, state it is not source of truth, and point back to the packet files. It must not become a second contract.

5. Phase anatomy — the same skeleton, mode-shaped

Required headings

Every phase file under 03-phases/*.md uses the same heading skeleton. The outcome and obligations headings shift with the mode; the rest is constant. A phase is a proof-gated slice, not a waterfall bucket.

## Safety/security constraints

  • live_provider_proof_blocker_only

  • visual_quality_gate (UI-bearing)
  • persistence_roundtrip
  • security_denied_path_test
  • no_fake_scan_pass

Hard rule: Phase mode contract

Every phase must declare blueprint_mode and phase_style matching the packet mirror. The lens explains why this phase uses that mode. The shared proof spine names preconditions, entrypoint, execution behavior, effects, observable proof, failure/recovery, and non-goals.

UI-bearing addendum

If the phase is UI-bearing (always true for product mode UI phases), proof requires a real user action path through UI → controller → runtime → visible/readback state. Default browser controls, stacked forms, generic cards, and screenshots-only proof do not satisfy this.

6. Role contracts under 06-contracts/

Six fixed roles

Each phase declares requires_roles: as a subset of these six. Role contracts are read only when the active phase requires them; they contain when active, handoff scope, reject if, required return headings, and proof/evidence expectations.

product-architect topology

Owns architecture for the active phase. Rejects local-MVP shortcuts and routes/route-shaped handlers.

  • First vertical slice
  • Component & dependency direction
  • ADR-lite tradeoffs

ux-ui-craft UI craft

Active on every UI-bearing phase. Owns the visual quality bar and Screenshot critique.

  • State quality (empty/loading/error/blocked/success)
  • Responsive + a11y
  • Domain affordances over generic dashboards

integration-runtime adapters

Provider/runtime/worker boundaries. Defines adapter seams, deterministic vs live modes, and blocker tracks.

  • Adapter / config / test wiring
  • Missing credentials block live proof only
  • Worker progress, retry, cancel, recovery

data-persistence durable state

State shape, migrations, restart/readback, lifecycle, retention, exports/deletes/resets.

  • Schema ownership & migrations
  • Restart/readback proof
  • Upload limits, retention, backup

security-boundary safety

Auth/session/tenant, secrets, destructive actions, denied paths, input validation.

  • Owner / tenant isolation
  • Denied-path tests
  • Destructive-action confirmation

test-and-verification evidence ceiling

Mandatory on every phase. Audits every upgrades_claim: true row and the proof ladder.

  • Evidence ceiling rule
  • No-fake scan & negative cases
  • Phase-state classification
Subagents optional; delegation artifacts mandatory.

When subagents / delegated workers are available, use them per role. When unavailable, the orchestrator self-simulates each required role and writes the same handoff / return / review artifacts. The artifacts are required even for a single-agent run.

8. Three-tier phase state model

Strictly hierarchical

These three states are non-interchangeable. Do not treat checkpoint_recorded as phase_core_passed; do not treat phase_core_passed as claim_qualified.

State What it proves What it does not prove Required artifact
checkpoint_recorded Evidence discipline: at least one valid runtime ledger row for the phase. Phase completion, UI quality, production readiness. At least one schema-valid ledger row in .buildprint/evidence/.
phase_core_passed Local vertical path works end-to-end with tests, reviews, and proof. For UI phases: a real user-action path. Live provider, browser e2e at scale, deployment, or qualified security claims. plan + team-gates + every handoff/return + 3 reviews + proof + scoped evidence.
claim_qualified A specific live/browser/visual/security/worker/deploy/data-lifecycle claim has matching executable proof. Any unrelated claim. One row never qualifies all tracks. A dedicated evidence row whose proves array matches its source exactly.

9. Evidence ceiling — can this row upgrade a claim?

Decision tree

Each row has a hard ceiling. Static text proves copy. Mocks prove contract shape, not live behavior. Reviews permit evidence; they do not upgrade claims. Use this tree before setting upgrades_claim: true.

?
Does the row’s command/artifact actually exercise the behavior in proves?
×
No.

The row is overclaiming. Narrow proves to what the source actually covers, or split into separate rows.

?
Yes → what kind of proof is this?
×
Review prose.

Reviews permit or block evidence; they never upgrade implementation proof by themselves. Set upgrades_claim: false.

×
Deterministic adapter / mock only.

Proves adapter seam & contract shape. Cannot upgrade live_provider_proof_*. Record a non-upgrading blocker if live is unavailable.

×
String / DOM-state check, screenshot only.

Cannot upgrade visual_quality_gate or ux_design_gate. Requires real browser e2e + critiqued screenshots.

×
HTTP smoke / aggregate test (npm test).

Proves an API path or build only. Cannot upgrade browser, worker, security, or data-lifecycle claims.

✓
Executable proof whose artifact / commands directly exercise the claim.

May set upgrades_claim: true. Keep proves narrow; use one row per behavior.

10. Continuation vs qualification

Two different gates

A blocker that prevents claim qualification does not automatically prevent downstream implementation. Mixing the two is how packets stall or fake green.

Continue (blocks_continuation: false) when

Core local implementation, API/domain/persistence tests, and required reviews are done. Only live-provider, browser/e2e tooling, deployment, or external-service blockers remain — recorded as honest non-upgrading rows.

Stop when

Core implementation failed, owned persistence is missing, local runtime/API proof failed, destructive/security ambiguity is unresolved, or evidence would be fake. Repair before continuing.

12. Anti-patterns — by mode

Mode-shaped failures

Each mode has its own dominant failure mode. The mapper rejects packets that exhibit these even when static checks pass.

productReject

Static UI shell as completion

Cards, labels, and dead buttons without a user action through controller → runtime → readback. Generic dashboards instead of domain affordances. Screenshots-only proof.

frameworklibraryReject

Framework collapsed into one app story

Phases written as user stories for an example app instead of primitives, invariants, composition rules, extension points, misuse states, and reference patterns.

integrationReject

Generic feature flow without boundary contract

No webhook/idempotency/secret/sandbox-vs-live split. Provider calls inline instead of through adapters with deterministic test doubles plus live-proof blockers.

automationReject

Feature checklist without loop & exit conditions

No task objective, tool/action boundary, plan/execute/observe loop, evidence requirements, stop conditions, approval points, or trace proof.

data-pipelineReject

Pipeline without lineage / validation / backfills

Transforms with no schema, no validation, no idempotent backfill, and no data quality proof. “Tests pass” without dataset-specific assertions.

infrastructureReject

Infra written as UI/API work

No deploy/apply entrypoint, resources changed, health, rollback, drift detection, observability, or permissions. The meaningful operation is the deployment, not a screen.

libraryReject

Library treated as one downstream app story

The callable API surface, semver/compat contract, and consumer patterns must be explicit. A single demo app does not substitute for primitive invariants, extension points, and misuse coverage.

mixedReject

“Mixed” used to avoid classifying

Mixed is valid when phases genuinely span lenses. Each phase must still declare its own blueprint_mode and phase_style; mixed is not a permission to be vague. If all phases would use the same mode, reclassify the packet under that mode.

all modesReject

Provider blocker as implementation escape

Missing credentials block live proof only. Adapter seams, config contracts, deterministic tests, and runtime wiring remain required regardless of mode.

13. Validator surface — what static checks enforce

Ground truth: check-mapper-selected-output.mjs

Static checks prove shape. They do not prove product quality. A green static check is necessary but never sufficient; replay and outcome judging remain required.
