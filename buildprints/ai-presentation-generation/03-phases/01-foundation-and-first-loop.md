# 01-foundation-and-first-loop

## How to implement this phase

Before writing code, read `03-phases/phase-flow.md`, `.buildprint/next-agent.md` if it exists, current project `AGENTS.md` if it exists, `BUILDPRINT.md`, `01-project-setup.md`, and `02-ui-identity.md` as the standing UI identity and user-language responsibility for every UI-bearing artifact. Build the smallest real local loop: create a presentation from prompt input, persist the presentation and ordered slide records, render an editable deck surface, and prove reload/readback. Use deterministic generation if live provider credentials are unavailable, but label it honestly and keep provider claims blocked.

## Building objective

Implement the product foundation as a working local webapp with backend persistence and a real slide workbench. The user should be able to open the app, pass setup/auth if enabled, enter a prompt, request a presentation, receive a coherent outline/deck generated by deterministic local logic or a configured provider seam, inspect ordered slides, select a slide, edit at least one content field inline, reorder or add a slide if that behavior is already dependency-ready, save automatically or explicitly, reload the page, and see the same deck restored from durable state.

This phase must not try to implement every provider or export path. It must establish the architecture that later phases extend: durable SQL records for presentation and slides, file storage directories, API endpoints with clear errors, frontend route flow, provider-mode labeling, and browser-visible state. The central output for this phase is not a perfect PPTX; it is a persisted editable deck that proves the app can turn user input into reviewable slide content without relying on source code or fake sample data.

## DO NOT

- Do not build a static sample deck and call it generation.
- Do not store presentations only in browser memory.
- Do not hide provider mode from the user.
- Do not create export buttons in this phase unless they are disabled with an honest blocker.
- Do not use raw JSON as the primary deck surface.
- Do not skip browser proof for the editor surface.
- Do not ship placeholders, lorem ipsum, empty wrappers, functionless buttons, inert navigation, swallowed errors, fake progress, or mocked/sample data counted as real proof.

## Minimum proof before moving on

- Backend test or smoke command creates, reads, updates, and deletes a presentation with ordered slides.
- Frontend/browser proof shows prompt entry, generated deck surface, selected slide, edit, save/readback after reload, and no fatal console errors.
- UI inspection confirms the slide canvas, thumbnail rail, and controls are not clipped or overlapping at desktop size.
- If provider calls are deterministic, the UI and handover say deterministic/local mode and do not claim live AI.
- Handover records the exact commands, URL, and any blocked provider/export claims.

## Handoff note

Record the first working deck fixture, database/storage paths, provider mode, browser URL, screenshots or inspection notes, and all missing live-provider/export evidence.
