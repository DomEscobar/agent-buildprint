## Phase mode contract

blueprint_mode: product
phase_style: outcome_flow

This phase is a product outcome flow over runtime data: users inspect the simulated world through action history, timeline, stats, posts, and comments with platform filters, pagination, empty states, and aggregate correctness.

## Product outcome

Turn OASIS runtime artifacts into an observable MiroFish world surface. Users can inspect what agents did, compare Twitter/Reddit activity, read posts/comments, see timeline and stats, and continue to report generation without using raw database or log files directly.

## Mapped product obligations

- Preserve paginated/filterable action history, action timeline, and per-agent stats.
- Preserve simulated Twitter posts and Reddit comments from per-platform SQLite or equivalent durable stores.
- Preserve browser simulation monitor/action feed states for running, completed, empty, filtered, failed, and dual-platform aggregate views.
- Preserve missing DB/log empty state and platform filtering behavior.

## Behavior compatibility contract

Target disposition: preserve/replace. Equivalent target behavior is required, not route/function parity. Storage engines and UI components can change, but action, post, comment, timeline, and stats read models must remain queryable, durable, filterable, and visible.

Compatibility impact:

- Replacing SQLite is allowed only if per-platform provenance, pagination/filter behavior, and missing-store empty states remain.
- Raw text-list substitutes do not preserve the product obligation; timeline, stats, and detail affordances must be domain-specific.
- Data generated by fake runtime fixtures can prove readback but cannot upgrade live OASIS provider claims.

## Implementation scope

Implement read models for actions, timeline, stats, posts, and comments; platform and agent filters; pagination; missing store behavior; aggregate correctness; browser timeline/detail UI; and fixture-backed tests that match expected OASIS action/post/comment shapes.

## Interfaces touched

- API/controller endpoints for action history, timeline, stats, posts, comments, platform filters, and pagination.
- Read repositories for action JSONL and per-platform DB/store outputs.
- Browser simulation observability panels, timeline, stats widgets, post/comment readers, filters, empty/error states.
- Tests for fixtures, missing files/DBs, invalid filters, pagination, and aggregate counts.

## State/runtime touched

Read but do not own OASIS runtime creation. Own normalized read models, pagination cursors, derived stats, timeline grouping, and empty/error readback behavior. Durable claims require persisted runtime artifacts from Phase 03 and readback tests in this phase.

## UX/UI requirements

This phase is UI-bearing. The surface must present actions, timeline, stats, posts, and comments as an inspection workspace with stable dimensions, filters, selected detail, empty/error states, and responsive behavior. UI proof needs repeatable_browser_e2e, visual_quality_gate, Screenshot critique, and no raw text-list substitute.

## Safety/security constraints

Redact private source text, provider tokens, and unsafe log content before display/evidence. Filter parameters must be validated. Missing DB/log paths must not leak server filesystem structure. Downloads remain Phase 07 unless explicitly scoped here.
