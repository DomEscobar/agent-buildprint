# Vision: Buildprint Mapper OS

Mapper OS should make AI builders better at reconstruction, not better at filling schemas.

The target output is a readable execution manual for a product-minded coding agent. It should teach the artifact identity, golden path, setup posture, UI/operator experience, phase flow, fake-success boundaries, and handover expectations.

## Why phase-driven

Phases are narrative and buildable. They let the mapper say: here is the coherent product slice, here is what to read, here is what to build, here is what not to fake, and here is what proves it.

The old model failed because tiny implementation documents compressed judgment into file names and runner mechanics. The new model keeps routing in YAML and puts real building knowledge in Markdown.

## Desired downstream behavior

A fresh coding agent should:

1. read the Buildprint in order;
2. answer only hard-stop questions;
3. create the setup foundation;
4. define UI/operator experience when relevant;
5. load only the active phase;
6. build the smallest real product path for that phase;
7. verify directly;
8. record a concise, honest handover.
