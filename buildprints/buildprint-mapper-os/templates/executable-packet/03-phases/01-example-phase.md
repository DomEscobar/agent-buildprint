# Phase 01 — First successful artifact loop

## Product intention

Create the smallest end-to-end path that lets the real consumer experience the artifact promise. For a product app, this is a user loop. For a developer-facing artifact, this is an adoption/configure/example loop. For a service or operator artifact, this is a transaction, recovery, or observability loop.

## Build

- The main input, command, API call, adapter, operation, or task path.
- The central artifact, work surface, public interface, boundary transaction, service state, or generated output.
- One meaningful interaction with that artifact or interface.
- The first useful output/result/trace.
- Local persistence or readback where the product needs continuity.
- Honest blocked states for unavailable live providers or external services.

## Quality bar

The intended consumer should understand what to do, see that their input/action changed the artifact behavior, and know the next action without reading the packet.

## Do not ship

Decorative artifact, generic dashboard, raw JSON dumped as the whole experience, dead controls, undocumented public methods, canned results, fake live-provider behavior, or framework seams that cannot be exercised.
