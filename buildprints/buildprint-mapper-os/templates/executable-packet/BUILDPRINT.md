# Buildprint Execution Start

This packet is an implementation input for a coding agent. Its job is to shape product judgment before coding, not to make the agent write proof-shaped prose after coding.

It is operated by the `agb` runner — see `templates/runner/RUNNER-SPEC.md`. The runner enforces persona injection, derived state, and mechanical drift checks so the agent cannot self-certify.

## Typed product-quality contract

You are the posture-aware product/developer/operator engineer for this run. First understand the artifact type and deployment posture declared in `blueprint.yaml`. Build the artifact the packet describes for its real consumer. Preserve the promise, identify the central artifact/boundary, make the first real loop usable, and challenge shallow implementation.

Do not optimize for the smallest artifact that satisfies wording. Do not create compliance theater. Passing tests means the app did not obviously break; it does not mean the product is good.

## Per-slice loop

For each slice declared in `slices/`:

1. Build session opens with `agb persona --slice slices/<id>/slice.yaml --role build` injected as system prompt. This loads the capsule from `templates/teams/`, the architecture summary, and the matching paths from `03-ux-contract.md` Path Map.
2. Builder implements the slice and writes `slices/<id>/slice-self-check.yaml` listing how each Block from the capsule was respected.
3. Acceptance session opens in a fresh agent with `agb persona --slice slices/<id>/slice.yaml --role acceptance`. The hostile-reviewer capsule rejects sample paths and blockers as evidence for `core_proof_required` paths.
4. Acceptance writes `slices/<id>/acceptance-result.json` with per-path `status`, `observed`, and `upgrades_claim`. A blocked core path forces `overall_slice_status: partial`, never `complete`.
5. `agb state derive` reads acceptance results and writes `.buildprint/state.json` (with `derived_by: "agb state derive"`).
6. `agb drift check` runs the tripwires; any failure blocks promotion of the slice.
7. `agb slice status` prints the human-readable report and updates `04-handover.md` from the template.

## Completion semantics

A slice is `complete` only when every path listed in `core_proof_required` has `upgrades_claim: true` in `acceptance-result.json`. Blocked, sample-satisfied, or stale-contract paths force `partial`. There is no OR-escape.

A gate is `passed` only when its result.json says so AND, if `requires_human_signoff: true`, `signoff_by` is non-agent.

Overall readiness is `complete` only when every slice is `complete` and every active gate is `passed`.

`trusted_local` posture lowers operability gates (security, observability, deploy) to `inactive`, never `passed`. It does not lower the product-craft floor for visible UI: a UI-bearing product at any posture uses a real component framework with a build step and design system.

Do not silently shrink scope. If something cannot be built, mark the slice `partial` with the specific path and reason.
