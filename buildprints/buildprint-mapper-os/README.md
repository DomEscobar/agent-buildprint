# Buildprint Mapper OS

Use this Buildprint to map a source project into a source-independent Buildprint package.

Mapper OS is run by an agent. It reads source, classifies the artifact type, distills mode-appropriate obligations, and produces either discovery output or a selected `selected-buildprint/` execution packet. The current direction is typed product-quality: the packet should improve downstream artifact quality for the real consumer, not maximize proof/evidence paperwork.

## Start Here

Read in this order:

1. `BUILDPRINT.md`
2. `SPEC.md`
3. `CONTRACTS.md`
4. `questions.md`
5. `policies/*.md`
6. `templates/runner/RUNNER-SPEC.md`

For generated selected packages, downstream implementers should not read all Markdown files upfront. They read: `BUILDPRINT.md`, `01-questions.md`, `02-project-setup.md`, `02-architecture.md`, `blueprint.yaml`, `03-ux-contract.md`, then only the active slice named in `slices/`.

## Default Behavior

- Discover first.
- Ask only blocking or implementation-changing questions.
- Treat scanner output as hints only.
- Keep output discovery-only until scope is selected.
- Preserve requested scope; do not hide broad, blocked, or unbuilt behavior.
- Produce executable Buildprint packets for selected outputs.
- Shape the downstream agent with posture-aware role/rule (`trusted_local`, `private_authenticated`, `public_webapp`): artifact type, real consumer, central artifact/interface/boundary, first usable loop, product-system spine where relevant, honest boundaries, forbidden shortcuts, skeptical review, concise handover.

## Selected Output (v2 Slice/Gate)

Selected output belongs under `selected-buildprint/`. Mapper OS output is a v2 Slice/Gate packet:

```text
BUILDPRINT.md
01-questions.md
02-project-setup.md
02-architecture.md
03-ux-contract.md
blueprint.yaml
slices/
  _template/
    slice.yaml
    build-brief.md
    acceptance-spec.md
  <id>/...
gates/
  gate-index.yaml
  <gate>.md
04-handover.md
```

`BUILDPRINT.md` is the execution start and read-order authority. `blueprint.yaml` is the machine contract. `02-architecture.md` defines stack and seams. `03-ux-contract.md` is the single Path Map truth (every slice references it). `slices/<id>/slice.yaml` declares `paths`, `core_proof_required`, and a `persona` reference into `templates/teams/`. `gates/gate-index.yaml` activates posture-conditional checks. `04-handover.md` is generated by `agb slice status` from the derived `state.json` — never authored manually.

Obsolete selected-output shapes are rejected: the v1 phase chain (`03-phases/`, `04-review.md`, `05-handover.md`, `generated/agent-prompt.md`), as well as `CAPABILITY_INDEX.md`, `CONTEXT_PACKET.json`, `TEAM_STACK.md`, `UX_CONTRACT.md`, `DESIGN_QUALITY_BAR.md`, `CURRENT_STATE.md`, `EXECUTION_PROTOCOL.md`, `IMPLEMENTATION_PLAN.md`, `manifest.json`, `02-context/active-slice.yaml`, `07-execution/phases/`, `capabilities/`, `05-evidence/`, `09-evidence/`, and per-capability mini-files.

## CLI Commands (agb)

The `bin/agb.js` CLI provides four runner commands for the Slice/Gate system:

```bash
# Build system prompt for a slice session (reads capsule from templates/teams/)
agb persona --slice slices/01-onboarding/slice.yaml --role build
agb persona --slice slices/01-onboarding/slice.yaml --role acceptance

# Derive state.json from acceptance-result.json files (never written by an agent directly)
agb state derive

# Run mechanical tripwires (path traceability, contract version, no fake provider, no self-write)
agb drift check

# Print human-readable slice + gate status report
agb slice status
```

## Qualification

Qualification is source-independent readiness for a downstream build. It is not proof that a future implementation already works. Built product quality must be judged from actual implementation output, not self-reported evidence prose.
